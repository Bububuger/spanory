# Spanory vs Langfuse OTLP Parity

## Source of Truth

Parity matrix is derived from Langfuse OSS ingestion and attribute constants:

- `/tmp/langfuse/packages/shared/src/server/otel/attributes.ts`
- `/tmp/langfuse/packages/shared/src/server/otel/OtelIngestionProcessor.ts`
- `/tmp/langfuse/packages/shared/src/server/otel/ObservationTypeMapper.ts`

## Scope

Runtime parsers (`@spanory/cli`):

- Claude Code (`claude-code`)
- OpenClaw (`openclaw`)

Transport: OTLP HTTP JSON payload generated by `compileOtlp`

Both runtimes share the same normalized event pipeline and OTLP compiler. Parity requirements are therefore
defined at normalized event level; runtime-specific gaps only apply when source transcript lacks required data.

## Gap Policy

Current policy for “100% compatibility”:

- If Langfuse field is derivable from transcript/runtime context: Spanory must emit it.
- If field is not derivable from transcript/runtime context: mark explicit gap with reason and keep schema extension point.

This means ingestion compatibility is complete for available data; unavailable fields are tracked and non-breaking.

## Runtime Coverage Notes

- `claude-code`: mature baseline used for initial parity implementation.
- `openclaw`: follows the same event categories and attribute schema as `claude-code`
  (`turn`, `agent_command`, `shell_command`, `mcp`, `agent_task`, `tool`), so Langfuse trace richness is aligned
  for derivable fields.
