# Spanory 修复计划：OpenClaw tool 细节可观测（2026-03-03 第二阶段）

## Goal
修复 OpenClaw realtime 上报中“session 有 turn 但缺少 tool 明细”的问题，确保 Langfuse 中每轮都能看到可用于排查的 tool 输入/输出细节。

## Root Cause
- OpenClaw embedded 路径的 `after_tool_call` context 缺失 `sessionKey/sessionId`，仅有 `toolName`，导致 runtime 侧无法稳定关联到正确会话。
- 现有实现主要依赖 `after_tool_call`/`tool_result_persist`；当 context 无法解析或 key 不一致时，tool 事件被直接丢弃。

## Scope
- In scope:
  - `packages/openclaw-plugin/src/index.js`
    - 新增 `before_message_write` 兜底采集：从持久化前消息中提取 assistant `toolCall` 与 `toolResult`
    - 增加 unresolved context 的安全兜底关联（最近活跃会话）
    - 保留现有 hook，但降低无 `toolCallId` 的低置信度重复上报
  - `packages/cli/test/unit/openclaw.plugin.runtime.spec.js`
    - 新增 `before_message_write` 路径的回归测试
- Out of scope:
  - 历史 trace 的离线修复
  - OpenClaw 上游 SDK 行为变更

## Tasks

### T1 增加失败用例（RED）
- 构造 `after_tool_call` 无会话上下文，仅有 `before_message_write` 的 `toolCall/toolResult` 场景。
- 断言工具 span 仍然生成，且挂到对应 turn。

### T2 实现 runtime 修复（GREEN）
- 增加 `onBeforeMessageWrite`：
  - 解析 assistant `toolCall`，缓存 `toolCallId -> params`
  - 解析 `toolResult`，恢复 tool span `input/output`
- 增加会话解析兜底：当 `requireResolved` 失败时回退到最近活跃会话。
- `after_tool_call` 在缺少 `toolCallId` 时不再主动上报（避免重复与误归属）。

### T3 验证与真实回归
- 跑 openclaw plugin runtime 单测。
- 在真实 OpenClaw 会话执行 1 次含 tool 的请求，确认 Langfuse 里出现 tool 细节。

## Acceptance
- 真实会话中，`turn` 与 `tool` 同时可见，且 tool 至少包含 `name`、`input`、`output`。
- 单测覆盖 `before_message_write` 兜底路径并通过。
