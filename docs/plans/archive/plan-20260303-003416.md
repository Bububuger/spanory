# Spanory 修复计划：OpenClaw 同 session 的 turn 冲突与 tool 漂移（2026-03-03）

## Goal
修复 OpenClaw realtime 上报中，长会话/重启后 `turnId` 重复导致 trace 合并错乱，以及 tool 回调晚到时挂错 turn 的问题，保证每轮可稳定看到对应 tool 细节。

## Root Cause
- 插件 runtime 使用进程内递增 `turn-<counter>`，网关重启后同一 session 会从 `turn-1` 重新开始，导致不同时间的轮次写入同一 trace。
- 当 `llm_output` 后按定时器提前 flush，若 `after_tool_call` 晚于 flush 到达，tool 事件进入 `pendingTools` 并在下一次 `llm_output` 被错误归属到下一轮。

## Scope
- In scope:
  - `packages/openclaw-plugin/src/index.js`：
    - turnId 生成改为跨重启低冲突（含时间分量）
    - 晚到 tool 直接归属当前 `lastTurnId`，不漂移到下一轮
  - `packages/cli/test/unit/openclaw.plugin.runtime.spec.js`：新增晚到 tool 回归测试
- Out of scope:
  - 历史已入库 trace 的数据修复（仅提供补救策略）
  - Langfuse UI 层改造

## Tasks

### T1 先补失败用例（RED）
- 构造“turn 已 flush，再收到 after_tool_call，再进入下一轮”的场景。
- 断言 tool 仍归属第一轮 turnId，而非第二轮。

### T2 最小修复 runtime 归属逻辑（GREEN）
- 引入 `createTurnId(state)`（基于输入时间戳 + 计数）替代纯计数 turnId。
- `onAfterToolCall` 在无 `currentBatch` 但有 `lastTurnId` 时，直接 submit 到该 turn。

### T3 验证与回归
- 跑 openclaw plugin runtime 单测。
- 用 session `2959b5c1-8730-4be8-a279-7ddbbeb96e39` 做服务端 observations 对账，确认 tool 可见性改善路径明确。

## Acceptance
- 同一 session 跨重启不再因 `turn-1` 重复发生 trace 合并错位。
- 晚到 tool 不再漂移到下一轮。
- 单测覆盖并通过。
