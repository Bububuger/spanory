# Spanory 计划：TypeScript 迁移前置基线（文档 + 测试）(2026-03-04)

## 目标
在“一次性从 JS 迁移到 TS”之前，先建立可验证的行为基线，确保重构后能力一致，尤其为后续桌面状态栏版本打下稳定契约基础。

## 成功标准
- 迁移契约文档明确：模块边界、输入输出契约、行为不变项、可破例项。
- 测试矩阵完整：每个关键模块至少映射到 1 组 unit/bdd 验证路径。
- 当前基线门禁全绿：
  - `npm run check`
  - `npm test`
  - `npm run test:bdd`
- 形成“可直接开工 TS 迁移”的入口清单（按模块优先级与风险排序）。

## 范围
### In Scope
- 计划与执行清单：`plan.md`、`todo.md`
- 文档产物：
  - `docs/plans/2026-03-04-ts-migration-contract.md`
  - `docs/plans/2026-03-04-ts-test-matrix.md`
  - `docs/plans/2026-03-04-ts-baseline-results.md`
- 基线验证命令执行与结果记录。

### Out of Scope
- 本阶段不做业务逻辑重构。
- 本阶段不改运行时上报语义。
- 本阶段不引入桌面端代码实现。

## 核心原则
1. 先锁行为，再改实现：BDD 结果是迁移后的行为真值。
2. 重构只允许“类型增强”，不允许未记录的行为变化。
3. 所有例外（若有）必须先写入迁移契约文档。

## 实施阶段
### 阶段 T0：阶段初始化
- 归档旧 `plan.md/todo.md`，写入本阶段新计划。

### 阶段 T1：迁移契约文档
- 明确各模块公共契约：
  - Event schema（`SpanoryEvent` / attributes）
  - Runtime adapter I/O
  - OTLP compile I/O
  - Plugin runtime side-effect（spool/retry/log）
- 定义“不变行为清单”与“允许变化清单（仅类型层）”。

### 阶段 T2：测试矩阵
- 建立模块到测试映射：`core/cli/otlp/plugin/report/alert`。
- 明确每条能力由哪些 unit/bdd 兜底。
- 标记缺口与补测优先级。

### 阶段 T3：基线跑测与记录
- 执行 `check/test/test:bdd`。
- 记录执行时间、文件数、测试数、失败数、潜在不稳定用例。

### 阶段 T4：迁移入口清单
- 按风险和依赖给出 TS 迁移顺序（模块级）。
- 给出每一批次的验收门禁与回滚点。

## 风险与缓解
- 风险：测试覆盖盲区导致迁移后行为漂移。
- 缓解：先补矩阵，再迁移；缺口先标红，不带病开工。
