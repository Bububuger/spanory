# Plan (2026-03-07) — 修复 Codex PTY Bash 耗时

## 背景
最新 Codex session 已确认使用当前本机构建的 `spanory`，但 ClickHouse 中最新 `Bash` 节点仍全部为 `duration_ms = 0`。导出 JSON 显示这些 `Bash` 事件只有约 `1ms`，说明问题不在 OTLP 编译，而在 Codex transcript 归一化阶段。

进一步定位表明，最新 Codex transcript 中 Bash 走的是 PTY 模式：
- `exec_command` 先返回 `session_id`
- 后续多次 `write_stdin(session_id=...)` 才表示同一工具持续运行

当前适配器把每个 `function_call_output` 都视为一次工具结束，导致 `exec_command` 只得到近乎同步的伪结束时间。

## 目标
- 修复 Codex PTY 模式下 Bash 工具耗时建模
- 让同一 PTY 会话在导出 JSON 与 ClickHouse 中都呈现非零持续时间
- 保持现有普通单次工具调用、Task、MCP 行为不回归

## 变更范围
- 文档流程：`plan.md`、`todo.md`
- Codex 适配器：`packages/cli/src/runtime/codex/adapter.ts`
- 测试：`packages/cli/test/unit/codex.adapter.spec.ts`

## 实施方案
1. 归档上一阶段 `plan.md/todo.md`，建立本阶段计划。
2. 先补一个失败测试，覆盖 `exec_command -> write_stdin(session_id) -> write_stdin(session_id)` 的 PTY 时长聚合。
3. 在 `codex/adapter.ts` 中为 PTY shell 会话建立 `session_id -> call` 关联，把后续 `write_stdin` 结果并入原始 Bash 调用，并用最后一次输出时间作为 `endedAt`。
4. 运行最小单测与本地 `export` 验证，再用 ClickHouse 对最新 session 复核。

## 验收标准
1. 新增测试先失败后通过，且覆盖 PTY 模式 Bash duration。
2. `spanory runtime codex export` 导出的最新 `Bash` 事件为非零持续时间。
3. ClickHouse 中最新 Codex trace 的 `Bash` observation 出现非零 `duration_ms`。
